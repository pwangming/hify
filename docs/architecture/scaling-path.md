# Hify 扩展路径：50 人 → 数千人

> 原则：**触发式演进，不做预防式建设**。每个阶段只解决被监控指标证实的瓶颈；
> 模块边界、API 契约、数据模型在全路径上保持不变——它们是 50 人阶段就买好的期权，
> 这份文档是行权时间表。

## 阶段总览

| 阶段 | 规模 | 一句话 | 预估工作量 |
|---|---|---|---|
| 0（现在） | 50~200 | 单机 4 容器，什么都不做 | — |
| 1 垂直扩容 | 200~500 | 换大机器 + 把监控立起来 | 1~2 天 |
| 2 水平扩展 | 500~1500 | 存算分离 + server 多副本 + 引入 Redis | 1~2 周 |
| 3 编排与抽离 | 1500~5000 | 上 K8s + workflow 执行器独立部署 | 2~4 周 |
| 4（按需） | >5000 | 视负载特征再议，大概率永远不需要 | — |

各阶段正交的一条线：**LLM 供应商额度**（见最后一节），它不随机器扩容自动解决。

---

## 阶段 1：垂直扩容 + 观测（200~500 人）

**触发条件（任一持续一周）**：CPU > 70%；JVM 老年代占用持续 > 80%；
API P95（非 LLM 链路）> 500ms；或用户数突破 200。

**改什么**：
- 机器 4C16G → 8C32G；同步调大：JVM 堆 4G→8G、HikariCP 20→40、
  postgres `shared_buffers`/`work_mem`、各供应商信号量；
- 把预留的监控立起来：Prometheus + Grafana 两个容器，盯四个核心指标——
  供应商 429 率、Hikari pending、CPU/堆、文档入库队列深度；
- pg_dump 挪到凌晨，错开使用高峰。

**不改什么**：零代码改动，拓扑不变。垂直扩容是最便宜的杠杆，先用完它。

---

## 阶段 2：存算分离 + 多副本（500~1500 人）

**触发条件**：垂直扩容到顶（16C 级别仍然吃紧）；或单点故障/发布窗口的业务影响
不可接受（几百人同时掉线已是事故）。

**改什么（按顺序，每步独立可回退）**：

1. **PostgreSQL 搬到独立机器**（或直接换云托管 RDS）。最先动存储是因为它解除单机资源
   争抢（HNSW 构建/备份不再抢 JVM 的 CPU），且只改连接串，风险最低。
2. **server 起 2~3 个副本**，nginx upstream 轮询。无状态 + JWT 早已就绪，副本本身零改动，
   但有三个配套必须做：
   - **引入 Redis（纯缓存身份）**：Spring Cache 实现从 Caffeine 切 Redis（配置级），
     每日 Token 配额计数也挪进 Redis（原子 INCR）；
   - **后台任务跨副本去重**：文档入库、workflow 异步执行改为 PG 任务表 +
     `FOR UPDATE SKIP LOCKED` 抢占式消费——这是阶段 2 唯一一段真正的新业务代码；
   - **优雅下线**：发布时副本先从 upstream 摘除 → 等在途 SSE 流和 workflow 收尾（已有
     graceful shutdown）→ 再停容器。滚动发布从此零感知。
3. **供应商信号量分布式化——用除法不用分布式锁**：单副本额度 = 总额度 ÷ 副本数。
   略有浪费但零复杂度；429 重试机制兜底误差。

**不改什么**：模块代码与边界、单体形态（仍是一个 jar）、不上 K8s（两三台机器
docker compose + 一个 nginx 完全管得住）、pgvector 不动。

---

## 阶段 3：K8s + 按角色拆部署（1500~5000 人）

**触发条件**：副本数 > 3 且手工管理发布/扩缩明显吃力；或 workflow 批量负载与对话交互
负载互相干扰（入库/工作流高峰拖慢对话 P95）；或公司有现成 K8s 平台与人手。

**改什么**：

1. **迁 K8s**：按 deployment.md 第 6 节映射表平移（nginx→Ingress、server→Deployment+HPA、
   sandbox→独立 Deployment + NetworkPolicy + gVisor、.env→Secret）。组件职责不变。
2. **workflow 执行器独立部署——拆部署，不拆代码**：同一个 jar，用 Spring profile 区分角色：
   `role=web`（只接 HTTP）和 `role=worker`（只消费任务表跑 workflow/入库）。
   两个 Deployment 独立扩缩，批量负载从此与交互负载物理隔离。
   **不需要拆仓库、不需要拆库、不需要 RPC**——模块化单体的"按角色部署"形态，
   这是当初坚持模块边界换来的最大一笔回报。
3. **sandbox 独立扩缩**：HPA 按执行队列深度，加 gVisor runtimeClass。
4. **数据层加固**：usage 写入改异步批量落库（事件机制已就绪，攒批 flush）；
   pgvector 按 dataset 分区 + 只读副本分担检索（触发条件：千万级 chunk 或检索 P95 > 100ms）。

**不改什么**：仍然不拆微服务——conversation/app/knowledge/provider 留在单体里；
PG 仍是唯一可信数据源；模块依赖规则与 `ModularityTests` 原样生效（它们不关心部署形态）。

---

## 阶段 4：>5000 人（按需，先别想）

只有当某个模块出现**独立的负载特征 + 独立的发版节奏 + 独立的团队**三个条件同时成立时，
才值得沿 Facade 边界把它抽成真服务（把门面实现换成 HTTP/队列客户端）。
最可能的候选依旧是 workflow；最不可能的是 conversation（和所有人耦合最深）。
在那之前，"单体多角色 + K8s 水平扩展"撑到万级用户没有结构性障碍。

---

## 贯穿全程：LLM 供应商额度线

机器可以加，供应商 RPM/TPM 不会自己长。各阶段对应动作：

| 阶段 | 动作 |
|---|---|
| 0~1 | 实测限额、对齐信号量、升供应商档位 |
| 2 | 同一供应商多 API Key 轮询（一期排除的"负载均衡"在此重新评估——此时是真需求） |
| 3 | 多供应商间按应用分流（app 配置指定模型本来就支持）；高频小模型场景评估本地推理（独立 GPU 机器，CLAUDE.md 既有约定） |

## 全程不变的清单（任何阶段动它们都要先回到本文档重新论证）

模块结构与依赖白名单、`api/` 门面契约、领域事件清单、PostgreSQL 单一可信源、
两协议模型接入、Flyway 迁移、前后端 monorepo、`code-organization.md` 全部规则。
